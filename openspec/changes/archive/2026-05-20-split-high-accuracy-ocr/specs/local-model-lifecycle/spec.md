## ADDED Requirements

### Requirement: Detect local model capability
The system SHALL detect whether the current device supports high-accuracy OCR. The system SHALL return `.supported` for Apple Silicon with >=16GB RAM, and `.unsupported` for all other cases (Intel architecture, Apple Silicon with <16GB RAM, or 0GB detected RAM). The 16GB minimum reflects peak inference memory usage of ~13GB for the full-precision model.

#### Scenario: Apple Silicon with 16GB RAM
- **WHEN** the device is Apple Silicon with 16GB unified memory
- **THEN** `DeviceCapabilityService.checkPaddleOCRCapability()` returns `.supported`

#### Scenario: Apple Silicon with 8GB RAM
- **WHEN** the device is Apple Silicon with 8GB unified memory
- **THEN** `DeviceCapabilityService.checkPaddleOCRCapability()` returns `.unsupported`

#### Scenario: Apple Silicon with 12GB RAM
- **WHEN** the device is Apple Silicon with 12GB unified memory
- **THEN** `DeviceCapabilityService.checkPaddleOCRCapability()` returns `.unsupported`

#### Scenario: Intel architecture
- **WHEN** the device is Intel architecture
- **THEN** `DeviceCapabilityService.checkPaddleOCRCapability()` returns `.unsupported`

#### Scenario: 0GB RAM detected (boundary)
- **WHEN** `ProcessInfo.physicalMemory` reports 0 bytes
- **THEN** `DeviceCapabilityService.checkPaddleOCRCapability()` returns `.unsupported`

### Requirement: Download local model on demand
The system SHALL allow users on devices with `.supported` local model capability to download the quantized OCR model from a configured HuggingFace URL to `~/Library/Application Support/MangaTranslator/Models/PaddleOCR-VL/`. The system SHALL report download progress. The system SHALL verify the downloaded file using SHA256 from a configured checksum source (for example `model.zip.sha256` generated by `upload.py`). The system SHALL persist download state and checksum in `UserDefaults`.

#### Scenario: Successful download
- **WHEN** user initiates download and the network request succeeds
- **THEN** model files are written to Application Support, SHA256 is verified against the configured checksum source, `UserDefaults` records `paddleocr.model.downloaded = true` and the checksum, and state transitions to `.downloaded`

#### Scenario: Secure extraction blocks path traversal
- **WHEN** a downloaded archive contains an entry whose resolved path escapes the model destination root
- **THEN** extraction fails, temporary artifacts are deleted, persisted download metadata is cleared, and state transitions to `.failed`

#### Scenario: Atomic install preserves prior model on failure
- **WHEN** extraction or final install fails after a previous valid model already exists
- **THEN** the previous model remains intact and usable, and state transitions to `.failed`

#### Scenario: Download with progress reporting
- **WHEN** download is in progress
- **THEN** `ModelDownloadService.state` is `.downloading(progress:)` with a value between 0.0 and 1.0

#### Scenario: SHA256 checksum mismatch
- **WHEN** download completes but the file SHA256 does not match the expected value
- **THEN** the partial/corrupt file is deleted, `UserDefaults` keys are cleared, and state transitions to `.failed` with a descriptive error

#### Scenario: Disk space insufficient
- **WHEN** the download directory has insufficient disk space
- **THEN** state transitions to `.failed` with a descriptive error message

#### Scenario: Network interrupted during download
- **WHEN** the network connection drops during download
- **THEN** state transitions to `.failed` with an error; the operation can be retried

#### Scenario: Download cancelled by user
- **WHEN** user cancels an in-progress download
- **THEN** any partial files are deleted and state transitions to `.notDownloaded`

#### Scenario: Duplicate download call
- **WHEN** `download()` is called while state is already `.downloading`
- **THEN** the second call is ignored; state remains `.downloading`

#### Scenario: Download directory not writable
- **WHEN** the Application Support directory cannot be written to
- **THEN** state transitions to `.failed` with a descriptive error

### Requirement: Verify local model integrity on app launch
The system SHALL verify model integrity on each app launch when `UserDefaults` indicates the model was previously downloaded. If the file is missing or the SHA256 does not match, the system SHALL reset state to `.notDownloaded` and clear `UserDefaults` keys. Launch-time verification SHALL execute asynchronously and SHALL NOT block initial UI rendering.

#### Scenario: File present and valid on launch
- **WHEN** app launches and model file exists with matching SHA256
- **THEN** state is `.downloaded`

#### Scenario: UserDefaults shows downloaded but file missing
- **WHEN** app launches and `paddleocr.model.downloaded` is `true` but model file does not exist
- **THEN** state resets to `.notDownloaded` and `UserDefaults` keys are cleared

#### Scenario: File present but SHA256 mismatch on launch
- **WHEN** app launches and model file exists but SHA256 does not match stored value
- **THEN** corrupt file is deleted, state resets to `.notDownloaded`, and `UserDefaults` keys are cleared

#### Scenario: Non-blocking launch verification
- **WHEN** app starts and checksum validation is still running
- **THEN** initial UI is rendered without waiting for checksum completion, and state updates after verification finishes

#### Scenario: Launch verification performs full checksum on stale integrity evidence
- **WHEN** stored checksum is missing/changed, model-version marker changed, prior verification failed, or model files are missing
- **THEN** `verifyOnLaunch()` performs full SHA256 verification before confirming `.downloaded`

#### Scenario: Launch verification fast-path when evidence is fresh
- **WHEN** checksum and integrity metadata are present and unchanged since the last successful verification
- **THEN** `verifyOnLaunch()` may use a fast-path integrity check without immediate full SHA256, while preserving eventual full verification policy

### Requirement: Delete local model
The system SHALL allow users to delete the downloaded model, freeing disk space. Deletion SHALL remove all files in the model directory, clear `UserDefaults` keys, disable the high-accuracy OCR preference, and transition state to `.notDownloaded`. If no file exists, `delete()` SHALL succeed silently without throwing.

#### Scenario: Delete existing model
- **WHEN** user confirms deletion and model files exist
- **THEN** all files under `Application Support/MangaTranslator/Models/PaddleOCR-VL/` are removed, `UserDefaults` keys are cleared, the high-accuracy OCR preference is set to `false`, and state becomes `.notDownloaded`

#### Scenario: Delete when file already absent
- **WHEN** `delete()` is called but model files do not exist
- **THEN** no error is thrown and state is `.notDownloaded`

#### Scenario: Delete while inference is in progress
- **WHEN** `delete()` is called while `PaddleOCRVLRecognizer` is actively running inference
- **THEN** deletion waits for the current inference to complete before removing files

#### Scenario: Delete and verify race
- **WHEN** `delete()` and `verifyOnLaunch()` overlap
- **THEN** model lifecycle coordination serializes state transitions so final state is deterministic and no partial model directory remains

### Requirement: Enforce deterministic local model state machine
The system SHALL enforce valid combinations of `ModelDownloadState` and `paddleocr.enabled` and SHALL converge to deterministic final states under concurrent operations.

Valid state combinations:

| `ModelDownloadState` | `paddleocr.enabled` | Valid |
| --- | --- | --- |
| `.notDownloaded` | `false` | yes |
| `.downloading(progress:)` | `false` | yes |
| `.downloaded` | `false` | yes |
| `.downloaded` | `true` | yes |
| `.failed(error)` | `false` | yes |
| any non-`.downloaded` | `true` | no |

Required transition rules:

| Event | From | To | Required side effects |
| --- | --- | --- | --- |
| User taps Download and Enable | `.notDownloaded + false` | `.downloading + false` | Start download; keep enable off until verified |
| Download+verify success | `.downloading + false` | `.downloaded + true` | Persist checksum and set `paddleocr.enabled = true` |
| Download fails | `.downloading + false` | `.failed + false` | Emit stable error code and keep enable off |
| User cancels download | `.downloading + false` | `.notDownloaded + false` | Remove partial artifacts |
| User enables manually | `.downloaded + false` | `.downloaded + true` | Reset recognizer |
| User disables manually | `.downloaded + true` | `.downloaded + false` | Reset recognizer |
| Delete model | `.downloaded + (true/false)` | `.notDownloaded + false` | Remove artifacts and reset recognizer |
| Verify finds missing/corrupt model | any `+ true` | corresponding non-downloaded state `+ false` | Clear invalid metadata before routing |

#### Scenario: Invalid enabled state is corrected
- **WHEN** `paddleocr.enabled` is `true` while model state is not `.downloaded`
- **THEN** the system resets `paddleocr.enabled` to `false` before OCR routing

#### Scenario: Concurrent delete and verify converge deterministically
- **WHEN** deletion and launch verification run concurrently
- **THEN** final state is deterministic, model artifacts are either fully present and valid or fully removed, and mixed partial state is not allowed
