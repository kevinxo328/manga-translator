## Purpose

Single owner for on-device local model lifecycle: device capability detection, model download from a configured remote source to Application Support, SHA256 integrity verification at install and at app launch, deletion, deterministic state convergence of `ModelDownloadState` and `paddleocr.enabled`, and invalid enabled-state correction. Other capabilities (`ocr-routing`, `settings-management`, `manga-ocr`, `paddleocr-recognizer`) consume this contract instead of restating it.
## Requirements
### Requirement: Detect local model capability
The system SHALL detect whether the current device supports high-accuracy OCR. The system SHALL return `.supported` for Apple Silicon with >=16GB RAM, and `.unsupported` for all other cases (Intel architecture, Apple Silicon with <16GB RAM, or 0GB detected RAM). The 16GB minimum reflects peak inference memory usage of ~13GB for the full-precision model.

#### Scenario: Apple Silicon with 16GB RAM
- **WHEN** the device is Apple Silicon with 16GB unified memory
- **THEN** `DeviceCapabilityService.checkPaddleOCRCapability()` returns `.supported`

#### Scenario: Apple Silicon with 8GB RAM
- **WHEN** the device is Apple Silicon with 8GB unified memory
- **THEN** `DeviceCapabilityService.checkPaddleOCRCapability()` returns `.unsupported`

#### Scenario: Apple Silicon with 12GB RAM
- **WHEN** the device is Apple Silicon with 12GB unified memory
- **THEN** `DeviceCapabilityService.checkPaddleOCRCapability()` returns `.unsupported`

#### Scenario: Intel architecture
- **WHEN** the device is Intel architecture
- **THEN** `DeviceCapabilityService.checkPaddleOCRCapability()` returns `.unsupported`

#### Scenario: 0GB RAM detected (boundary)
- **WHEN** `ProcessInfo.physicalMemory` reports 0 bytes
- **THEN** `DeviceCapabilityService.checkPaddleOCRCapability()` returns `.unsupported`

### Requirement: Download local model on demand
The system SHALL allow users on devices with `.supported` local model capability to download the quantized OCR model from a configured HuggingFace URL. The system SHALL report download progress. The system SHALL verify the downloaded archive using SHA256 from a configured checksum source (for example `model.zip.sha256` generated by `upload.py`).

The system SHALL preserve legacy installs at `~/Library/Application Support/MangaTranslator/Models/PaddleOCR-VL/`. New successful installs SHALL use `~/Library/Application Support/MangaTranslator/Models/PaddleOCR-VL.current/` as the active directory. The service SHALL use the parent directory `~/Library/Application Support/MangaTranslator/Models/` as the install container for `.current`, `.installing`, and backup directories.

The system SHALL perform every new install as an atomic transaction:

- download the archive to a temporary file outside any active model directory
- verify archive checksum before extraction
- create a unique staging candidate at `<models>/.installing/PaddleOCR-VL.next.<uuid>`
- store the verified archive at `<models>/.installing/PaddleOCR-VL.next.<uuid>/model.zip`
- extract archive contents only into the staging candidate
- reject extraction if `/usr/bin/unzip` exits with non-zero status
- reject extraction if any resolved extracted path escapes the staging candidate
- validate the staging candidate model structure before changing the active directory
- rename `.current` to `PaddleOCR-VL.backup.<uuid>` only when `.current` exists and the staging candidate is already valid
- rename the staging candidate to `PaddleOCR-VL.current`
- delete backup after the new `.current` is installed successfully

The system SHALL NOT move, delete, overwrite, or modify an existing valid `PaddleOCR-VL.current` or legacy `PaddleOCR-VL` directory before the staging candidate has passed checksum verification, secure extraction, and model structure validation. The system SHALL persist `paddleocr.model.downloaded`, `paddleocr.model.checksum`, and `paddleocr.model.lastVerified` only after `PaddleOCR-VL.current` exists and resolves as valid.

Failure state rules SHALL be deterministic. If a failed install attempt started with a previously resolved valid model, the system SHALL keep the prior downloaded/checksum metadata, keep the prior enabled preference, and return to `.downloaded` so high-accuracy OCR remains usable. If a failed install attempt started without a previously resolved valid model, the system SHALL clear downloaded/checksum metadata, set `paddleocr.enabled = false`, and transition to `.failed`.

#### Scenario: Successful download with no prior model
- **WHEN** user initiates download, the network request succeeds, checksum matches, extraction succeeds, and the staging candidate validates
- **THEN** the system installs the model at `PaddleOCR-VL.current`, stores `model.zip` inside `PaddleOCR-VL.current`, records `paddleocr.model.downloaded = true` and the checksum in `UserDefaults`, and transitions state to `.downloaded`

#### Scenario: Successful update with existing current model
- **WHEN** a valid `PaddleOCR-VL.current` exists and a new download completes checksum verification, extraction, and staging validation
- **THEN** the system renames the old `PaddleOCR-VL.current` to a unique backup, renames the valid staging candidate to `PaddleOCR-VL.current`, deletes the backup, updates downloaded/checksum metadata, and future resolution uses the new `PaddleOCR-VL.current`

#### Scenario: Successful update with existing legacy model
- **WHEN** a valid legacy `PaddleOCR-VL` exists, `PaddleOCR-VL.current` does not exist, and a new download completes checksum verification, extraction, and staging validation
- **THEN** the system installs the new model at `PaddleOCR-VL.current`, leaves the legacy `PaddleOCR-VL` directory unchanged, updates downloaded/checksum metadata, and future resolution prefers `PaddleOCR-VL.current`

#### Scenario: Secure extraction blocks path traversal
- **WHEN** a downloaded archive contains an entry whose resolved path escapes the staging candidate directory
- **THEN** extraction fails, staging artifacts for the current attempt are deleted, prior valid `.current` or legacy model directories remain unchanged, and the final state follows the deterministic failure state rules

#### Scenario: Non-zero unzip exit fails install
- **WHEN** `/usr/bin/unzip` exits with any non-zero status during model extraction
- **THEN** extraction fails, staging artifacts for the current attempt are deleted, prior valid `.current` or legacy model directories remain unchanged, and the final state follows the deterministic failure state rules

#### Scenario: Atomic install preserves prior current model on failure
- **WHEN** extraction, staging validation, active rename, or final install fails after a previous valid `PaddleOCR-VL.current` already exists
- **THEN** the previous current model remains intact or is restored from backup, remains usable by resolver, downloaded/checksum metadata remains unchanged, enabled preference remains unchanged, and state returns to `.downloaded`

#### Scenario: Atomic install preserves prior legacy model on failure
- **WHEN** extraction, staging validation, active rename, or final install fails while only a valid legacy `PaddleOCR-VL` model exists
- **THEN** the legacy model remains intact, remains usable by resolver, downloaded/checksum metadata remains unchanged, enabled preference remains unchanged, and state returns to `.downloaded`

#### Scenario: First install failure has no prior model
- **WHEN** checksum verification, extraction, staging validation, active rename, or final install fails and no valid model resolved before the attempt started
- **THEN** artifacts created by the current attempt are deleted, downloaded/checksum metadata is cleared, `paddleocr.enabled` is set to `false`, and state transitions to `.failed`

#### Scenario: Rollback failure is visible
- **WHEN** `.current` was renamed to backup and rollback from backup to `.current` also fails
- **THEN** the system logs the rollback failure through the debug logging facility, does not modify the legacy root, does not mark the failed attempt as downloaded, and the final state follows the deterministic failure state rules

#### Scenario: Download with progress reporting
- **WHEN** download is in progress
- **THEN** `ModelDownloadService.state` is `.downloading(progress:)` with a value between 0.0 and 1.0

#### Scenario: SHA256 checksum mismatch
- **WHEN** download completes but the file SHA256 does not match the expected value
- **THEN** the partial/corrupt archive is deleted, staging is not promoted to active, prior valid `.current` or legacy model directories remain unchanged, and the final state follows the deterministic failure state rules with a descriptive error available for first-install failure

#### Scenario: Disk space insufficient
- **WHEN** the download directory has insufficient disk space
- **THEN** prior valid `.current` or legacy model directories remain unchanged, and the final state follows the deterministic failure state rules with a descriptive error available for first-install failure

#### Scenario: Network interrupted during download
- **WHEN** the network connection drops during download
- **THEN** the operation can be retried, prior valid `.current` or legacy model directories remain unchanged, and the final state follows the deterministic failure state rules

#### Scenario: Download cancelled by user
- **WHEN** user cancels an in-progress download
- **THEN** artifacts created by the current attempt are deleted, prior valid `.current` or legacy model directories remain unchanged, state returns to `.downloaded` if a valid model resolved before the attempt started, and state transitions to `.notDownloaded` if no valid model resolved before the attempt started

#### Scenario: Duplicate download call
- **WHEN** `download()` is called while state is already `.downloading`
- **THEN** the second call is ignored; state remains `.downloading`

#### Scenario: Download directory not writable
- **WHEN** the Application Support model container cannot be written to
- **THEN** prior valid `.current` or legacy model directories remain unchanged, and the final state follows the deterministic failure state rules with a descriptive error available for first-install failure

### Requirement: Verify local model integrity on app launch
The system SHALL verify model integrity on each app launch when `UserDefaults` indicates the model was previously downloaded. Launch-time verification SHALL resolve the active model directory using the local model resolver. If `PaddleOCR-VL.current` exists and is valid, launch verification SHALL verify `<models>/PaddleOCR-VL.current/model.zip` against the stored checksum. If `.current` does not exist, launch verification SHALL support the legacy archive path `<models>/PaddleOCR-VL/model.zip` when the legacy root resolves as valid.

If the resolved model directory is missing, has no stored archive, lacks supported model weights, or fails SHA256 verification, the system SHALL reset state to `.notDownloaded`, clear `UserDefaults` download/checksum keys, and set `paddleocr.enabled = false`. Launch-time verification SHALL execute asynchronously and SHALL NOT block initial UI rendering.

#### Scenario: Current model present and valid on launch
- **WHEN** app launches, `PaddleOCR-VL.current` resolves as valid, `PaddleOCR-VL.current/model.zip` exists, and the archive checksum matches the stored checksum
- **THEN** state is `.downloaded`

#### Scenario: Legacy model present and valid on launch
- **WHEN** app launches, `.current` does not exist, legacy `PaddleOCR-VL` resolves as valid, `PaddleOCR-VL/model.zip` exists, and the archive checksum matches the stored checksum
- **THEN** state is `.downloaded`

#### Scenario: UserDefaults shows downloaded but resolved model missing
- **WHEN** app launches and `paddleocr.model.downloaded` is `true` but no valid model directory resolves
- **THEN** state resets to `.notDownloaded`, `UserDefaults` download/checksum keys are cleared, and `paddleocr.enabled` is set to `false`

#### Scenario: Resolved archive missing on launch
- **WHEN** app launches and a model directory resolves but the expected `model.zip` inside that resolved directory does not exist
- **THEN** state resets to `.notDownloaded`, `UserDefaults` download/checksum keys are cleared, and `paddleocr.enabled` is set to `false`

#### Scenario: File present but SHA256 mismatch on launch
- **WHEN** app launches and the resolved archive SHA256 does not match stored checksum
- **THEN** state resets to `.notDownloaded`, `UserDefaults` download/checksum keys are cleared, and `paddleocr.enabled` is set to `false`

#### Scenario: Non-blocking launch verification
- **WHEN** app starts and checksum validation is still running
- **THEN** initial UI is rendered without waiting for checksum completion, and state updates after verification finishes

#### Scenario: Launch verification performs full checksum on stale integrity evidence
- **WHEN** stored checksum is missing/changed, model-version marker changed, prior verification failed, or model files are missing
- **THEN** `verifyOnLaunch()` performs full SHA256 verification before confirming `.downloaded`

#### Scenario: Launch verification fast-path when evidence is fresh
- **WHEN** checksum and integrity metadata are present and unchanged since the last successful verification
- **THEN** `verifyOnLaunch()` may use a fast-path integrity check without immediate full SHA256, while preserving eventual full verification policy

### Requirement: Delete local model
The system SHALL allow users to delete the downloaded model, freeing disk space. Deletion SHALL remove `PaddleOCR-VL.current`, legacy `PaddleOCR-VL`, `.installing`, and any `PaddleOCR-VL.backup.<uuid>` directories under the model container. Deletion SHALL clear `UserDefaults` keys, disable the high-accuracy OCR preference, and transition state to `.notDownloaded`. If no file exists, `delete()` SHALL succeed silently without throwing.

#### Scenario: Delete existing current model
- **WHEN** user confirms deletion and `PaddleOCR-VL.current` exists
- **THEN** `PaddleOCR-VL.current`, `.installing`, and all `PaddleOCR-VL.backup.<uuid>` directories are removed, `UserDefaults` keys are cleared, the high-accuracy OCR preference is set to `false`, and state becomes `.notDownloaded`

#### Scenario: Delete existing legacy model
- **WHEN** user confirms deletion and legacy `PaddleOCR-VL` exists
- **THEN** legacy `PaddleOCR-VL`, `.installing`, and all `PaddleOCR-VL.backup.<uuid>` directories are removed, `UserDefaults` keys are cleared, the high-accuracy OCR preference is set to `false`, and state becomes `.notDownloaded`

#### Scenario: Delete when file already absent
- **WHEN** `delete()` is called but no current, legacy, staging, or backup model artifacts exist
- **THEN** no error is thrown and state is `.notDownloaded`

#### Scenario: Delete while inference is in progress
- **WHEN** `delete()` is called while `PaddleOCRVLRecognizer` is actively running inference
- **THEN** deletion waits for the current inference to complete before removing files

#### Scenario: Delete and verify race
- **WHEN** `delete()` and `verifyOnLaunch()` overlap
- **THEN** model lifecycle coordination serializes state transitions so final state is deterministic and no partial model directory remains

### Requirement: Resolve active local model directory
The system SHALL resolve the active local model directory deterministically from the model container. The model container is the parent directory of the legacy `config.modelDirectory`.

The resolver SHALL check candidates in this exact order:

1. `<models>/PaddleOCR-VL.current`
2. `<models>/PaddleOCR-VL`
3. the single valid model child under `<models>/PaddleOCR-VL`

A candidate SHALL be valid only when it contains supported model weights according to the existing supported model predicate (`weights.npz` or `model.safetensors`). If no candidate is valid, resolution SHALL return no model. If the legacy root contains more than one valid child model and neither `.current` nor the legacy root itself is valid, resolution SHALL return no model rather than choosing arbitrarily.

#### Scenario: Current directory wins over legacy root
- **WHEN** both `<models>/PaddleOCR-VL.current` and `<models>/PaddleOCR-VL` contain supported model weights
- **THEN** resolver returns `<models>/PaddleOCR-VL.current`

#### Scenario: Legacy root is used when current is absent
- **WHEN** `<models>/PaddleOCR-VL.current` does not exist and `<models>/PaddleOCR-VL` contains supported model weights
- **THEN** resolver returns `<models>/PaddleOCR-VL`

#### Scenario: Legacy single child is used when root has one valid child
- **WHEN** `.current` does not exist, legacy root does not directly contain supported model weights, and legacy root contains exactly one child directory with supported model weights
- **THEN** resolver returns that child directory

#### Scenario: Ambiguous legacy children do not resolve
- **WHEN** `.current` does not exist, legacy root does not directly contain supported model weights, and legacy root contains more than one child directory with supported model weights
- **THEN** resolver returns no model

#### Scenario: Invalid current falls back to valid legacy root
- **WHEN** `<models>/PaddleOCR-VL.current` exists but lacks supported model weights and `<models>/PaddleOCR-VL` contains supported model weights
- **THEN** resolver returns `<models>/PaddleOCR-VL`

### Requirement: Clean model install artifacts
The system SHALL remove install artifacts created by the current attempt after successful installs and handled failures. Install artifacts include `<models>/.installing/PaddleOCR-VL.next.<uuid>` and any empty `<models>/.installing` directory created by the current attempt. After a successful install, the system SHALL also remove the backup directory created for that install.

The system SHALL NOT remove legacy `PaddleOCR-VL` as part of install cleanup. The system SHALL NOT remove a prior valid `PaddleOCR-VL.current` unless the new valid staging candidate has already been promoted or rollback has restored the prior current directory.

#### Scenario: Successful install leaves no staging or backup
- **WHEN** a new model is successfully installed to `PaddleOCR-VL.current`
- **THEN** the staging candidate for that attempt no longer exists, the backup directory for that attempt no longer exists, and `PaddleOCR-VL.current` resolves as valid

#### Scenario: Failed extraction cleans staging
- **WHEN** extraction fails because of path traversal or non-zero `unzip` exit status
- **THEN** the staging candidate for that attempt no longer exists and prior valid `.current` or legacy model directories remain unchanged

#### Scenario: Failed validation cleans staging
- **WHEN** extraction succeeds but the staging candidate lacks supported model weights
- **THEN** the staging candidate for that attempt no longer exists, prior valid `.current` or legacy model directories remain unchanged, and the final state follows the deterministic failure state rules

### Requirement: Enforce deterministic local model state machine
The system SHALL enforce valid combinations of `ModelDownloadState` and `paddleocr.enabled` and SHALL converge to deterministic final states under concurrent operations.

Valid state combinations:

| `ModelDownloadState` | `paddleocr.enabled` | Valid |
| --- | --- | --- |
| `.notDownloaded` | `false` | yes |
| `.downloading(progress:)` | `false` | yes |
| `.downloaded` | `false` | yes |
| `.downloaded` | `true` | yes |
| `.failed(error)` | `false` | yes |
| any non-`.downloaded` | `true` | no |

Required transition rules:

| Event | From | To | Required side effects |
| --- | --- | --- | --- |
| User taps Download and Enable | `.notDownloaded + false` | `.downloading + false` | Start download; keep enable off until verified |
| Download+verify success | `.downloading + false` | `.downloaded + true` | Persist checksum and set `paddleocr.enabled = true` |
| Download fails | `.downloading + false` | `.failed + false` | Emit stable error code and keep enable off |
| User cancels download | `.downloading + false` | `.notDownloaded + false` | Remove partial artifacts |
| User enables manually | `.downloaded + false` | `.downloaded + true` | Reset recognizer |
| User disables manually | `.downloaded + true` | `.downloaded + false` | Reset recognizer |
| Delete model | `.downloaded + (true/false)` | `.notDownloaded + false` | Remove artifacts and reset recognizer |
| Verify finds missing/corrupt model | any `+ true` | corresponding non-downloaded state `+ false` | Clear invalid metadata before routing |

#### Scenario: Invalid enabled state is corrected
- **WHEN** `paddleocr.enabled` is `true` while model state is not `.downloaded`
- **THEN** the system resets `paddleocr.enabled` to `false` before OCR routing

#### Scenario: Concurrent delete and verify converge deterministically
- **WHEN** deletion and launch verification run concurrently
- **THEN** final state is deterministic, model artifacts are either fully present and valid or fully removed, and mixed partial state is not allowed

### Requirement: High-accuracy OCR settings section
The system SHALL display a "High-Accuracy OCR" section in SettingsView exclusively on Apple Silicon (`#if arch(arm64)`). The section SHALL be hidden entirely on Intel. The section SHALL reflect the current `ModelDownloadService.state` and update reactively as state changes.

#### Scenario: Intel device
- **WHEN** the app runs on an Intel Mac
- **THEN** the High-Accuracy OCR section is not visible in Settings

#### Scenario: Apple Silicon, not downloaded
- **WHEN** the device is Apple Silicon and the model has not been downloaded
- **THEN** the section shows a "Download and Enable" button

#### Scenario: Apple Silicon 8GB, not downloaded
- **WHEN** the device is Apple Silicon with 8GB RAM and the model has not been downloaded
- **THEN** the section shows the "Download and Enable" button and a warning label about RAM

#### Scenario: Download in progress
- **WHEN** model download is in progress
- **THEN** the section shows a progress indicator and a "Cancel" button; "Download and Enable" is hidden

#### Scenario: Model downloaded and enabled
- **WHEN** the model is downloaded and high-accuracy OCR is enabled
- **THEN** the section shows an enabled indicator, a "Disable" button, and a "Delete Model Data" button

#### Scenario: Model downloaded and disabled
- **WHEN** the model is downloaded and high-accuracy OCR is disabled
- **THEN** the section shows a disabled indicator, an "Enable" button, and a "Delete Model Data" button

### Requirement: Confirm before deleting model data
The system SHALL present a confirmation dialog before deleting the model. The dialog SHALL use English text. Deletion SHALL only proceed after user confirmation.

#### Scenario: User confirms deletion
- **WHEN** user clicks "Delete Model Data" and then confirms in the dialog
- **THEN** `ModelDownloadService.delete()` is called and state transitions to `.notDownloaded`

#### Scenario: User cancels deletion
- **WHEN** user clicks "Delete Model Data" but cancels in the confirmation dialog
- **THEN** no deletion occurs and state is unchanged

### Requirement: Persist high-accuracy OCR preference
The system SHALL persist the user's high-accuracy OCR enabled/disabled preference in `UserDefaults` under the key `paddleocr.enabled`. State-machine constraints on this preference â€” rejecting enable when the model is not `.downloaded`, and resetting to `false` on deletion â€” follow the "Enforce deterministic local model state machine" and "Delete local model" requirements in this capability; Settings UI surfaces those decisions to the user. Recognizer-reset on preference change is owned by `ocr-routing`'s "Reset recognizer on engine switch" requirement.

#### Scenario: Preference persists across launches
- **WHEN** user enables high-accuracy OCR and relaunches the app
- **THEN** high-accuracy OCR remains enabled if the model is still present

#### Scenario: Enable blocked when model is absent
- **WHEN** the model is not downloaded (or fails verification) and user attempts to enable high-accuracy OCR
- **THEN** the transition is rejected per the state machine table (this capability's "Enforce deterministic local model state machine" requirement), Settings keeps the disabled/not-downloaded state, and the UI displays actionable guidance ("Download model first")

#### Scenario: Enable blocked after failed verification
- **WHEN** model verification fails and user attempts to enable high-accuracy OCR
- **THEN** enable is rejected per the state machine, an error message explains model integrity failure, and the UI offers re-download guidance
